Replace the spreadsheet without introducing the opposite failure mode: numbers that look plausible but can't be audited.

Financial analysts run valuation models — DCF, comparables, sensitivity tables — manually in spreadsheets. The workflow is slow to iterate on, impossible to version, and entirely disconnected from the research context that informs the assumptions. Replacing it with an LLM introduces the opposite failure mode: models that produce numbers that look plausible but aren't deterministically reproducible and can't be audited back to their inputs.

A finance-grade system needs both: AI-assisted research and interpretation, with deterministic computation underneath it that an analyst can verify line by line. The hard part is enforcing that separation architecturally — not as a comment in the code, but as a boundary the LLM physically cannot cross.

Four specialised agents, strict Pydantic boundaries, zero LLM involvement in the computation path.

  • The Research Agent interprets the user's analytical intent and generates structured context for downstream agents. The Data Engineering Agent validates and normalises incoming financial datasets against Pydantic schemas before they reach the computation layer — a malformed payload fails loudly at the boundary, not silently inside the model.
  • The Finance Agent runs deterministic DCF analysis, IRR, and scenario modelling in Python — no LLM involvement at this stage. All inputs are Pydantic-validated before the calculation runs. All outputs are Pydantic-validated before leaving the service. Output artifacts are versioned as CSV for downstream compatibility with Excel and BI tools.
  • The Research Agent then receives the validated numerical output and generates written investment narrative grounded in the model's results — it reasons over the numbers, not around them. Citations in the narrative reference the actual model output fields, not LLM-generated summaries of them.
  • n8n acts as the external control plane, orchestrating the agent sequence, triggering data fetches, handling retry logic and error routing, and assembling the final report. FastAPI exposes the computation layer as a service that n8n calls — orchestration and computation stay decoupled.

Why these choices — each enforces the AI/computation separation as a hard constraint, not a soft guideline.

  • LLMs for narrative only — deterministic Python for all math. Financial calculations (DCF, IRR, sensitivity tables) are deterministic Python with typed Pydantic models. LLMs are invoked only to interpret results and generate written analysis — never to produce numbers. This isn't a guardrail; it's an architectural constraint enforced by agent boundaries. The LLM has no access to the computation layer, only to its validated output.
  • n8n for workflow orchestration over custom DAG code. The data fetch, model trigger, and report assembly steps change frequently as the analytical workflows evolve. A visual DAG in n8n is faster to iterate on than Python scheduler code, makes the workflow auditable without reading source, and handles retry logic and error routing without custom implementation.
  • Pydantic for strict inter-agent I/O contracts. Every financial model input and output is a Pydantic schema. This acts as a firewall between agents — if upstream data is malformed, or a data source returns an unexpected structure, the pipeline fails fast at the boundary before bad data contaminates downstream calculations. Schema validation is the enforcement mechanism for the LLM/computation separation: the LLM can only receive what passed validation.
  • MOCK mode as a first-class execution mode. All LLM calls are substituted with deterministic responses in MOCK mode — making the pipeline fully testable without API keys or spend. The Finance Agent computation path runs identically in both modes; only the Research Agent's LLM calls are swapped. CI tests run against MOCK mode exclusively.

What broke during build — and what I changed to fix it.

  • The LLM narrative drifted from the model output when the prompt was too open-ended. An early prompt asked the Research Agent to "summarise the DCF analysis". It produced confident-sounding narrative that paraphrased — and occasionally contradicted — the actual model numbers. The fix: the prompt explicitly enumerates every field from the validated output and requires the narrative to reference each one by name. The LLM interprets; it cannot invent.
  • n8n retry logic amplified a Pydantic validation error into duplicate model runs. When the Data Engineering Agent returned a validation error, n8n's default retry policy retried the whole workflow — including the FastAPI computation call that had already completed. The fix: FastAPI computation endpoints are idempotent by request ID; n8n retries the data validation step only, not the full pipeline.
  • Sensitivity table output was too large for the LangChain context window at default chunk size. A 10×10 sensitivity table serialised to JSON exceeds 2k tokens — the Research Agent's context window was filling with sensitivity data before the DCF summary arrived. The fix: sensitivity tables are serialised as compact CSV (not JSON) and truncated to the 3×3 centre slice for narrative purposes; the full table is always in the versioned CSV artifact.
  • MOCK mode diverged from PROD mode in a non-obvious way. The MOCK Research Agent returned a fixed string rather than a string that referenced the actual model output fields — so MOCK tests passed even when the prompt was broken. The fix: MOCK responses are generated by a template that interpolates the actual validated output fields, making them functionally equivalent to a real LLM call for testing purposes.
