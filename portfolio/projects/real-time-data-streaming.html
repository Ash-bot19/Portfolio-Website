Batch pipelines are useless when you need to see a spike the moment it happens.

Event-driven systems generate data at rates that outpace what batch pipelines can handle — a Grafana dashboard refreshing every 5 minutes is useless when you need to see a processing spike the moment it happens. The challenge isn't just ingestion speed: it's building a topology that handles variable-rate bursts without backpressure failures, guarantees exactly-once processing across Spark micro-batches on restart, and separates the streaming path from the batch path so neither blocks the other.

Orchestrating Kafka, Spark, and Cassandra into a coherent Lambda architecture with minimal operational overhead — with four Airflow DAGs handling cross-system dependencies — is the core engineering problem.

Lambda architecture with clean streaming/batch separation and cross-system DAG orchestration.

  • Airflow DAGs ingest events from external APIs into Kafka topics partitioned by event type — isolating consumer lag per stream so a slow consumer on one topic doesn't affect others. Retention is configured to allow full replay on consumer failure.
  • Spark Structured Streaming micro-batches consume from Kafka, apply windowed transformations and aggregations (tumbling and sliding windows), checkpoint state to persistent storage for exactly-once delivery on restart, and write to Cassandra with time-based partition keys tuned for the time-series read pattern.
  • The batch layer mirrors selected events into PostgreSQL for Airflow-scheduled reporting jobs. Grafana reads directly from Cassandra for real-time panels — the streaming and batch paths are completely isolated consumers on the same Kafka topic.
  • Four main DAGs handle the end-to-end flow: API-to-Kafka ingestion, Kafka-to-Cassandra streaming health monitoring, batch-to-Postgres loading, and Cassandra compaction monitoring. Airflow's DAG model handles cross-system dependency management cleanly — Cron can't express these dependencies.

Why these choices — each driven by a concrete failure mode or scale constraint.

  • Cassandra over PostgreSQL for event storage. Write-heavy time-series workload with no complex joins — Cassandra's partition-key model gives linear write scale without the overhead of a relational engine. The compaction strategy is tuned specifically for the time-series read pattern, keeping read latency flat as the dataset grows. PostgreSQL is retained for the batch layer where join semantics matter.
  • Spark Structured Streaming over Kafka Streams. Kafka Streams is JVM-only and handles stateful aggregations over short windows cleanly, but Spark's Python API unlocks the broader ML and data engineering ecosystem. Spark also handles larger stateful windows more naturally and integrates directly with the existing Airflow orchestration layer for checkpoint management.
  • Airflow for pipeline orchestration over custom cron. The pipeline has genuine dependencies between stages — the Cassandra compaction task must run after streaming jobs complete, and the Grafana data-source refresh depends on compaction finishing. Cron can't express those dependencies. Airflow's DAG model handles cross-system dependency management cleanly.
  • Time-based Cassandra partition keys. A naive key design on a write-heavy time-series workload creates hotspots — all writes land on the same partition until the TTL rotates. Time-bucketed partition keys (hourly or daily depending on throughput) distribute writes across the cluster without manual resharding.

What broke in production — and what I changed because of it.

  • Consumer lag is a lagging indicator — it hides the real problem. Early versions monitored Kafka consumer lag and triggered alerts when it crossed a threshold. But lag only grows after the problem has persisted for multiple micro-batch intervals. The fix: instrument Spark's processing time per micro-batch and alert on that directly — lag is a consequence, not a cause.
  • Cassandra partition hotspots don't show up until you exceed one partition's write throughput. On small datasets, the partition key strategy looks fine — all reads and writes complete fast. At scale, a poorly chosen partition key routes all writes to a single node, causing that node's CPU to spike while others sit idle. The fix required profiling the actual write distribution with nodetool and redesigning the partition key to include a bucket suffix.
  • Spark checkpoint state grows unboundedly without an explicit cleanup policy. An early version didn't configure state TTL for aggregations — after a week of continuous operation, checkpoint state occupied several GB of disk and Spark's recovery time on restart grew proportionally. The fix: explicit watermarks on event-time aggregations so Spark can expire state that's past the watermark.
  • Airflow's Kafka consumer DAG can fall behind during a spike and never recover without manual intervention. The DAG was polling Kafka at a fixed interval. Under a sustained spike, each DAG run fell further behind the partition head. The fix: expose consumer lag as an Airflow metric and trigger a backfill branch in the DAG when lag exceeds a threshold rather than relying on the fixed-interval schedule.
